## Lessons Learned

Captured during the plan-pipeline build (PLAN 202605011400 + children, 2026-05-01). Architectural, process, and user-preference observations worth preserving.

### Architectural

- **Mechanical and conceptual review are different operations needing different model tiers.** Six rounds of Sonnet-grade mechanical check-plan against the parent PLAN returned `Blockers: 0` consistently. A Human-led Opus pass after those surfaced six real architectural issues (load-bearing assumption verification, bootstrap validation timing, dogfood fidelity, orchestrator edge cases, memory-note staleness, decision-overhead meta). Lesson: a passing plan-safety check is NOT a sufficiency check. They use different lenses; they benefit from different model tiers. Hence the split into `audit-sufficiency` (opus) and `audit-haiku-safe` (sonnet). Both must pass before a PLAN advances.

- **Skills can't introspect their execution context.** "Refuse if invoked from a subagent" is unimplementable — there's no documented API. The actual safeguard for parent-session-only skills (like `ideate`) is structural: don't create an agent file in `.claude/agents/` pointing to the skill. Absence-of-agent-file IS the safeguard.

- **Subagents do NOT inherit the parent's skill registry at runtime.** Documented Claude Code behaviour. Workaround: each agent file declares `skills: [skill-name]` in frontmatter, which preloads the skill content at startup. Dynamic `Skill("foo")` calls from inside a subagent fail.

- **Mid-execution streaming and external cancellation of background subagents are not documented as supported.** Don't design around them. The transactional return-on-completion IS the contract.

- **Subagent-driven exception handling beats orchestrator-driven supervision** — Erlang's "let it crash" pattern. Subagent self-terminates on declared `<exception_conditions>`, returns `outcome: exception` + diagnostics, orchestrator decides whether to kanban full-stop. Cleaner than trying to monitor/cancel from outside.

- **Wire format ≠ durable record.** The agent's return message is the wire format (`<pipeline-result>` JSON block — greppable, parseable, fails-loud-on-malformed). PLAN frontmatter fields (`pipeline_phase`, `audit_state`, `last_executor_outcome`, `verification_state`) are the durable disk record. Wire is for immediate parse; disk is for re-entry idempotency. Conversation history is too fragile to rely on for state.

- **State assertions and spec acceptance sampling are different verification tiers** (decision 25). `verify:` checks "the pieces are in place"; `acceptance:` exercises "does it actually do what the Objective claims?" Both shell-runnable; zero LLM tokens on the happy path. Catches "executor said done but reality disagrees" — a class of failure that an LLM-grade per-step verification (current Step 3 of execute-plan workflow) is vulnerable to via confirmation bias.

- **Triage decisions before asking the Human** (decision 15). Don't dump 25 decisions and ask "still locked?" — classify Already-locked / Mechanically-forced / Real-judgement-call; surface only the third class. Saves Human attention; the other two classes get listed once for transparency.

- **Bootstrap chicken-and-egg is unavoidable for self-bootstrapping infrastructure.** A PLAN that builds an orchestrator can't be orchestrated by that orchestrator (it doesn't exist yet). Accept manual bootstrap; future PLANs use the new infrastructure.

### Process

- **Focused agent probes catch fatal architectural assumptions early.** Probe-1 (verify pinned-model agents, subagent skill registry, run_in_background semantics) caught two would-be-fatal mistakes before any code was written — subagent skill registry isolation, and undocumented mid-execution streaming. Cost: one Agent call. Saved: weeks of unwinding.

- **Multiple Opus review passes are not redundant.** Each pass after a major design change can surface issues mechanical review can't. Six mechanical rounds + three Opus rounds during the parent PLAN's drafting; the Opus rounds were where decisions 19, 21, 22, 23, 24, 25 emerged. Cheap relative to the cost of building the wrong thing.

- **PLAN-as-design-spec emerged organically.** When we accepted that "create a new skill" is design work (not mechanical execution), the PLANs in that category became acceptance criteria rather than Haiku-executable scripts. Authoring proceeds by Opus directly; PLAN's Verification serves as the contract. The bootstrap children (1410, 1420, 1425, 1430) are this category. Future PLANs from the orchestrator-built pipeline can be properly Haiku-safe.

- **Re-running check-plan after every design change** kept the design's blocker-count visible. The hard-gate semantics (decision 14) also forced honest tallies — couldn't hand-wave past sufficiency issues.

### User preferences (Ken)

- **Options-with-recommendation, always.** When there's a real choice, present 2–3 options in full with tradeoffs, then state the lean and explain why. Single-option-as-fait-accompli reads as you skipping work.

- **Push back when I'm overengineering.** Ken interrupted multiple times when the design was growing complexity for limited benefit. Listen for "this misses the point", "i think we are concentrating on X when we can do Y instead", or quick course-corrections like "ignore the stub framing then, sorry".

- **Ken's metaphors are signals.** "Pull the kanban full-stop lever" was a real architectural directive (subagent-driven exception → orchestrator decides whether to halt). Don't treat them as decorative.

- **Catches model-context drift.** "we're still on opus, whoops" — Ken corrected my model context multiple times. Always confirm the active model when the conversation has been long; mention it explicitly when switching.

- **Token economy matters more than feature breadth.** Ken pushed back on chatty-subagent-with-milestones design when probe-2 found platform support fragile. Accepted "subagent returns once with outcome enum" as a feature (transactional boundary), not a limitation.

- **Codified behaviour beats memory crutches.** Ken's "let's remove this memory file. we'll have the orchestrator git push according to its stepwise logic" — when behaviour can be encoded into the orchestrator's deterministic logic, prefer that over a feedback memory note that future Claudes might misread.

- **Verify spec, not just steps.** Ken pushed back on my conflation of state-assertion verification with spec-verification. "the actual result must be verified against the spec. even if only a sample of it." Decision 25's two-tier `verify:` + `acceptance:` came from this push.

- **Fast feedback loops over long autonomous runs.** Ken's pacing was many short interjections ("ok", "fix first then stop", "pause after this round"). Don't barrel through; offer natural break points.

### Conventions clarified during this work

- Skills declare contracts in XML-tagged blocks: `<preconditions>`, `<inputs>`, `<output_schema>`, `<exception_conditions>`, `<constraints>`, `<success_criteria>` (consistent with existing `<essential_principles>` etc.)
- Subagents preload skills via `skills: [...]` frontmatter; never dynamic-dispatch
- `background: true` reserved for subagents that genuinely need parent-responsiveness (= currently only `plan-executor` variants)
- Outcome enum is exactly three values: `success | revision_needed | exception`
- Orchestrator dispatches by `(phase, outcome)` tuple — exhaustive routing table
- Audit loops use per-stage iteration counters with `MAX_ITERATIONS=5` to prevent runaways
- Wire format: `<pipeline-result>` opener + JSON code fence + closing tag at end of agent's response (decision 23)
- Pre-existing files (April-month PLAN modifications, `Bus/202605010230_*`) are intentionally not touched by this work — out of scope
- Memory: `feedback_retire_push.md` deleted because retire-skill-decoupling supersedes it; orchestrator's stepwise commit logic (decision 22) is the new source of truth
