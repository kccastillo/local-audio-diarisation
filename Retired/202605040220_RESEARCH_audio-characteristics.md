## Why this exists

After committing v2 to a WhisperX-based architecture without alternatives analysis, the operator paused execution and asked for proper research. This file captures the *concrete characteristics* of the two test recordings the operator named, so the architecture decision is grounded in the actual audio rather than in library marketing.

## Files measured

Three recordings were measured in `recordings/`. Direct measurement via `scripts/inspect_audio.py` (uses `librosa` + `audioread`). A reference Teams-generated transcript is also available for the third file (`.docx` companion).

**Update 2026-05-04**: the operator deleted **File 2** (`meeting 30 Apr at 1-34 pm.m4a`) after this measurement run, confirming it was not recorded properly. The analysis below predicted exactly that — File 2's anomalous spectral centroid (7.4 kHz vs typical 1.5-3 kHz), 75 % silence, and inverted SNR profile all pointed at upstream-damaged audio. Useful as a recorded data point for what "broken input" looks like — pipeline gates would block such a file at Gate 2 (voice presence) before it consumed any GPU time. File 1 and File 3 remain.

## What each file actually is

**File 1 — typical "live meeting" recording.** Loud, near-clipping peaks, ~30 dB SNR, mostly speech. Probably a laptop mic in an actual room with multiple humans. This is what we'd hope for.

**File 2 — heavily processed, sparse.** A 50 dB SNR with a noise floor near digital silence is not natural — it points at upstream processing. The 7.4 kHz spectral centroid is the giveaway: human speech typically centroids at 1.5–3 kHz; pushing it past 7 kHz means *something has stripped mid-band content*. The likely culprit is an AI-based denoiser/voice-isolator (NVIDIA Broadcast, Krisp, Discord-Krisp, or a misconfigured Teams). Combined with 75 % silence, this looks like a remote / streamed-source recording where:

- Long silences reflect single-talker turn-taking on a conference call where only one mic was active at a time.
- Aggressive AI denoise has been applied somewhere in the pipeline before this file was saved.
- Mid-band suppression may have damaged speech intelligibility (unverifiable without listening; the centroid alone strongly suggests it).

**File 2 is harder for Whisper than its 50 dB SNR suggests.** SNR is a power-domain metric; if the speech itself has been spectrally degraded, Whisper will struggle even though SNR looks pristine.

**File 3 — Microsoft Teams recording, the operator's "typical" case.** This is the canonical input for the v2 use case (operator confirmed 2026-05-04). Already 16 kHz mono, RMS at a strong −20.9 dBFS, peak just under full-scale, voiced 86 %, spectral centroid at a textbook 1 754 Hz. Teams' upstream processing has gated non-speech to digital zero (which is why noise floor reads as −∞), but — critically — the speech-band content is intact. This is the *easy* case for the pipeline. A reference transcript exists in the matching `.docx`, generated by Teams' own STT — useful as a quality benchmark for v2 output. Sample of the reference: full speaker labels (e.g. "Kenneth Castillo 0:03"), correct technical vocabulary ("RITMs", "macro block", "MVP"), readable prose. Teams' transcript quality is the bar v2 needs to clear or beat.

## Implications for the pipeline

These three files together describe the operating envelope:
- **File 3 (Teams)** is the typical case: well-prepared, 16 kHz mono, no preprocessing needed. Pipeline should detect this and short-circuit preprocessing.
- **File 1 (live mic)** is the harder common case: 48 kHz stereo, near-clipping, mostly speech. Needs format conversion only.
- **File 2 (denoised remote)** is the edge case: quiet, sparse, spectrally damaged. Needs adaptive treatment and the operator should expect lower transcription quality.

This motivates a pipeline that **measures input characteristics before deciding what to do**, rather than applying a fixed filter chain. Specific consequences:

1. **Conditional format conversion.** If input is already 16 kHz mono PCM-equivalent (File 3), pass through unchanged. Otherwise resample to 16 kHz mono (Files 1, 2). Whisper-family models want 16 kHz mono.

2. **Adaptive loudness normalisation, not blind.** File 1 sits at −26.7 dBFS (good); applying +10 dB loudnorm would push it into clipping. File 2 sits at −41.2 dBFS (too quiet); leaving it alone hurts model performance. Decision rule: measure RMS post-decode, normalise only when RMS is below a threshold (e.g. −32 dBFS).

3. **Adaptive denoising, default OFF.** File 1's 30 dB SNR is fine for Whisper. File 2 has *already been denoised* — running another denoiser will further damage speech detail. Decision rule: skip denoise unless input noise floor is above −40 dBFS *and* SNR is below 20 dB. Most meeting recordings won't meet that threshold.

4. **VAD-based silence trimming is high-value for File 2.** Cutting 75 % of silence reduces transcription compute by ~75 % on File 2. Cheap (Silero VAD runs in seconds on a 30-min file) and conservative.

5. **Validation gates between stages.** Before committing to a 30+ minute Whisper run, fail fast on:
   - Decode failure → halt with clear error.
   - Empty / near-silent file (voiced fraction < 1 %) → halt; refuse to spend Whisper time on silence.
   - Catastrophic preprocessing damage (post-SNR drops by >3 dB or RMS drops by >6 dB after denoise) → revert to pre-denoise audio.
   - 30-second dry-run transcription on the first chunk → fails fast if Whisper can't even detect language.
   - Diarisation produces sensible speaker count → warn if 1 detected with no `--no-diarisation` flag (likely a diarisation failure on a multi-talker file) or if speaker count exceeds a sane bound.

6. **Spectral-centroid sanity check.** If centroid > 6 kHz, log a warning that the input has likely been through aggressive denoise; transcription quality may be degraded vs raw audio. This is an *informational* gate — it doesn't block, but it gives the operator a heads-up if the result is worse than expected.

7. **Stereo handling.** Both files are stereo. Default to channel-mean for mono. *Do not* discard one channel — if there are participants on different channels (some conference systems do this), averaging preserves all voices.

## What this audio data does NOT tell us

- Whether the speakers are well-diarised by current models — diarisation quality is a function of the model, not the input metrics. Empirical run needed.
- Whether the cybersecurity jargon transcribes well — vocabulary fit is a Whisper-variant question (large vs medium vs distil), not an input-characteristics question.
- Whether 8 GB VRAM is enough — purely a function of model + compute_type + batch size, independent of input.

## Caveats on these measurements

- "Estimated SNR" here is the difference between the median-of-top-10 % and median-of-bottom-10 % frame-RMS, which is a coarse proxy. It won't catch subtle stationary noise, and it overstates SNR when the noise floor is already digital silence (as in File 2).
- "Voiced fraction" uses an energy threshold (noise_floor + 6 dB), not a real VAD. It's a sanity number, not a diarisation input.
- "Spectral centroid" is a single scalar averaged over the whole file. A more useful diagnostic would be the centroid distribution across speech-only frames, but the single-number version is enough to flag File 2's anomaly.
